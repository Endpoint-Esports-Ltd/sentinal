# /sync - Sync Project Rules & Skills

**Sync custom rules and skills with the current codebase.** Reads existing rules/skills, explores code patterns, identifies gaps, updates documentation, creates new skills.

**Flow:** Read existing → Migrate → Quality audit → Explore → Compare → Sync project/MCP/skills → Discover rules/skills → Cross-check → Summary

**Team sharing:** Commit `.claude/` to source control and share via normal PR workflows.

---

## Phase 0: Reference

### Guidelines

- **Always use AskUserQuestion** when asking the user anything
- **Read before writing** — check existing rules before creating
- **Write concise rules** — every word costs tokens in context
- **Idempotent** — running multiple times produces consistent results

### Project Slug

Derive the project slug from the git repo or directory name. Use it as a prefix for ALL created rules and skills to avoid name collisions across repositories.

```bash
# Derive slug: git repo name (preferred) or directory basename
SLUG=$(basename "$(git remote get-url origin 2>/dev/null | sed 's/\.git$//')" 2>/dev/null || basename "$PWD")
# Result: "my-api", "acme-backend", "sentinal"
```

Use `{slug}-` prefix on everything: `{slug}-project.md`, `{slug}-mcp-servers.md`, `{slug}-{topic}.md`, `.sentinal/skills/{slug}-{name}/`.

### Output Locations

**Custom rules** in `.sentinal/rules/`: `{slug}-project.md` (tech stack, structure), `{slug}-mcp-servers.md` (custom MCP servers), `{slug}-{pattern-name}.md` (tribal knowledge).

**Custom skills** in `.sentinal/skills/`: `{slug}-{name}/SKILL.md`.

Both are read by Claude Code and OpenCode via symlinks and config (set up by `sentinal install`).

Use unique names (not `plan`, `implement`, `verify`, `standards-*`) for custom skills.

### Recommended Directory Structure

**For monorepos with multiple products/teams**, organize rules in nested subdirectories:

```
.sentinal/rules/
├── README.md                          # Structure guide (generated by /sync)
├── {slug}-project.md                  # Repo-wide: tech stack, structure, commands
├── {slug}-mcp-servers.md              # MCP server documentation
├── {slug}-conventions.md              # Repo-wide coding conventions
├── {product-a}/                       # Product A rules
│   ├── {slug}-{product-a}-project.md  # Product-wide: stack, patterns, commands
│   ├── {slug}-{product-a}-api.md      # Product-wide: API conventions
│   ├── team-x/                        # Team X rules (MUST use `paths`)
│   │   └── {slug}-team-x-data.md      # Team-specific patterns
│   └── team-y/                        # Team Y rules (MUST use `paths`)
│       └── {slug}-team-y-frontend.md  # Team-specific patterns
├── {product-b}/                       # Product B rules (same structure)
│   └── ...
└── common/                            # Shared libraries rules
    └── {slug}-common-utils.md         # Shared utility conventions
```

**Scoping rules:**

| Directory Level                | `paths` Frontmatter | Loaded When                    |
| ------------------------------ | ------------------- | ------------------------------ |
| Root (`.sentinal/rules/*.md`)    | Optional            | Every session (keep lean)      |
| Product (`{product}/*.md`)     | Recommended         | Files matching product paths   |
| Team (`{product}/{team}/*.md`) | **MANDATORY**       | Files matching team paths only |

**⛔ Team-level rules without `paths` frontmatter load every session for every file** — defeating the purpose of scoping. Always add `paths` to team-level rules.

**Example `paths` frontmatter:**

```yaml
---
paths:
  - "src/ubervoice/**"
  - "tests/ubervoice/**"
---
```

**When to use nested structure vs flat:** If the repo has 2+ distinct products, teams, or major modules with different conventions → nested. If it's a single-product repo → flat is fine.

### Error Handling

| Issue                        | Action                                                                  |
| ---------------------------- | ----------------------------------------------------------------------- |
| Vexor not installed          | Use Grep/Glob for codebase exploration (all Vexor steps have fallbacks) |
| `/learn` skill not available | Create skills manually (write SKILL.md directly)                        |
| No MCP servers               | Skip MCP documentation                                                  |
| No README.md                 | Ask user for description                                                |
| No package.json              | Infer from file extensions                                              |

### Writing Rules

Rules load every session — every word costs tokens.

- **Lead with the rule** — what to do first, why second
- **Code examples over prose** — show, don't tell
- **Skip the obvious** — don't document standard framework behavior
- **One concept per rule** — don't combine unrelated patterns
- **Bullet points > paragraphs** — scannable beats readable
- **Max ~100 lines per file** — split large topics
- **Meaningful descriptions** — never write vague comments like "Also an entry". State what it does and when it's used.
- **Portable paths** — use relative paths or variables, not absolute paths that break on other machines
- **Current syntax** — use modern tool versions and command syntax (e.g., `docker compose` not `docker-compose`)

### Claude Code Best Practices Reference

**What belongs in rules (loaded every session):**

- Bash commands Claude can't guess
- Code style rules that differ from defaults
- Testing instructions and preferred test runners
- Repository etiquette (branch naming, PR conventions)
- Architectural decisions specific to the project
- Developer environment quirks (required env vars)
- Common gotchas or non-obvious behaviors

**What does NOT belong in rules:**

- Anything Claude can figure out by reading code
- Standard language conventions Claude already knows
- Detailed API documentation (link to docs instead)
- Information that changes frequently
- Long explanations or tutorials
- File-by-file descriptions of the codebase
- Self-evident practices like "write clean code"

**Structure & size:**

- Target under 200 lines per file — longer files reduce adherence
- Use markdown headers and bullets — organized sections > dense paragraphs
- Write instructions concrete enough to verify: "Use 2-space indentation" not "Format code properly"
- Remove conflicting instructions — contradictions cause arbitrary behavior
- Use emphasis ("IMPORTANT", "YOU MUST") for critical rules

**Scoping:**

- Use `paths` frontmatter in `.sentinal/rules/` to scope rules to specific file types
- Domain knowledge or workflows only relevant sometimes → skills (loaded on demand), not rules (loaded every session)

**Litmus test for every instruction:** "Would removing this cause Claude to make mistakes?" If not, cut it.

## Phase 0.5: Migrate Legacy Paths to `.sentinal/`

**Run BEFORE Phase 1.** Check if project has rules/skills in legacy locations (`.claude/rules/`, `.claude/skills/`, `.opencode/rules/`, `.opencode/skills/`) that are real directories (not symlinks). If found, migrate to `.sentinal/` and create symlinks.

```bash
# Check for real directories (not symlinks) that need migration
if [ -d ".claude/rules" ] && [ ! -L ".claude/rules" ]; then echo "MIGRATE: .claude/rules → .sentinal/rules"; fi
if [ -d ".claude/skills" ] && [ ! -L ".claude/skills" ]; then echo "MIGRATE: .claude/skills → .sentinal/skills"; fi
if [ -d ".opencode/rules" ] && [ ! -L ".opencode/rules" ]; then echo "MIGRATE: .opencode/rules → .sentinal/rules"; fi
if [ -d ".opencode/skills" ] && [ ! -L ".opencode/skills" ]; then echo "MIGRATE: .opencode/skills → .sentinal/skills"; fi
```

**If any migrations needed:**

1. `mkdir -p .sentinal/rules .sentinal/skills`
2. Copy contents from legacy dirs to `.sentinal/` (skip if destination already exists — when both `.claude/rules/` and `.opencode/rules/` exist with the same filenames, keep the first copy found)
3. Remove the legacy directories
4. Create symlinks: `.claude/rules → ../.sentinal/rules`, `.claude/skills → ../.sentinal/skills`, `.opencode/skills → ../.sentinal/skills`
5. Report: "Migrated N rules and M skills to .sentinal/"

**If no migrations needed:** Skip silently, proceed to Phase 1.

## Phase 1: Read Existing Rules & Skills

**MANDATORY FIRST STEP.**

1. Derive the project slug (see Phase 0 → Project Slug)
2. `find .sentinal/rules/ -name '*.md' -not -name 'README.md' 2>/dev/null | sort` — read each rule file (including subdirectories)
3. `ls -la .sentinal/skills/*/SKILL.md 2>/dev/null` — read each skill file
4. Check for legacy CLAUDE.md: `ls CLAUDE.md claude.md .claude.md 2>/dev/null` — read if found
5. **Detect unscoped legacy files** — look for `project.md`, `mcp-servers.md`, or any rule/skill without the `{slug}-` prefix. Flag for migration in Phase 2.
6. **Detect nested rule directories** — check for subdirectories within `.sentinal/rules/`. Map each subdirectory, its depth level (product vs team), and contents.
7. **Validate path-scoping** — for rules in nested subdirectories, check that they have `paths` frontmatter. Flag any team-level rules (depth 2+) missing `paths` as errors for Phase 3.
8. Build inventory: documented rules (with directory structure map), skills, CLAUDE.md contents, gaps, outdated items, legacy files, nested directories, path-scoping violations

## Phase 2: Migrate Unscoped Assets — CONDITIONAL

**Only if Phase 1 found unscoped files.**

AskUserQuestion: "Found unscoped assets that should be prefixed with '{slug}-' for better sharing. Migrate now?"

- **"Yes, migrate all"** — rename each to `{slug}-{name}`, update internal references, delete old file
- **"Review each"** — show each file, let user decide per-file
- **"Skip"** — leave as-is, continue sync

**Migration rules:**

- `project.md` → `{slug}-project.md` | `mcp-servers.md` → `{slug}-mcp-servers.md` | `{topic}.md` → `{slug}-{topic}.md`
- `.sentinal/skills/{name}/` → `.sentinal/skills/{slug}-{name}/` (update `name:` in frontmatter too)
- Legacy `.claude/skills/` or `.opencode/skills/` → move to `.sentinal/skills/`
- Do NOT migrate files from `~/.claude/rules/` — those are global user rules

## Phase 3: Quality Audit

**Audit all existing rules and CLAUDE.md files against Claude Code best practices (Phase 0 → Best Practices Reference).** Present findings as improvement suggestions — do NOT modify files without user confirmation.

**Skip this phase if:** No existing rules or CLAUDE.md files found in Phase 1.

### Step 3.1: Run Checks

For each rule file and CLAUDE.md found in Phase 1, evaluate:

| Check                   | What to look for                                                                          | Severity |
| ----------------------- | ----------------------------------------------------------------------------------------- | -------- |
| **Size**                | Files over 200 lines (context bloat, reduced adherence)                                   | Warning  |
| **Specificity**         | Vague instructions: "format properly", "write clean code" — suggest concrete alternatives | Warning  |
| **Redundancy**          | Standard conventions Claude already knows                                                 | Info     |
| **Conflicts**           | Contradicting instructions across different files                                         | Error    |
| **Path-scoping**        | Rules mentioning specific file types but lacking `paths` frontmatter                      | Info     |
| **Nested path-scoping** | Rules in team-level subdirectories without `paths` frontmatter                            | Error    |
| **Skills candidates**   | Multi-step workflows in rules better as skills (on demand vs every session)               | Info     |
| **Structure**           | Dense paragraphs without headers/bullets, poor scanability                                | Warning  |
| **Stale references**    | References to files, commands, or tools that no longer exist                              | Error    |
| **CLAUDE.md overlap**   | Content duplicated between CLAUDE.md and `.sentinal/rules/` files                           | Warning  |

**How to check for stale references:**

```bash
# File paths
ls -la <referenced-path> 2>/dev/null

# Patterns via Vexor (preferred) or Grep fallback
vexor search "<referenced-pattern>" ./ --max-results 1 --max-tokens 500
# Fallback: Grep(pattern="<referenced-pattern>", head_limit=5)
```

### Step 3.2: Present Findings

Group findings by severity and present to user:

```
## Quality Audit Results

### Errors (should fix)
- ❌ **Conflict:** rule A says "use tabs" but CLAUDE.md says "use 2-space indent"
- ❌ **Stale:** project.md references `src/legacy/` which no longer exists

### Warnings (recommended)
- ⚠️ **Size:** CLAUDE.md is 340 lines — split into modular rules (target: <200 lines)
- ⚠️ **Vague:** style.md: "Format code properly" → suggest: "Run `prettier --write` before committing"

### Suggestions (nice to have)
- 💡 **Path-scope:** testing.md only mentions `*.test.ts` — add `paths: ["**/*.test.ts"]` frontmatter
- 💡 **Skill candidate:** deployment.md contains a 40-line workflow — convert to a skill
```

### Step 3.3: User Decision

AskUserQuestion (multiSelect): "Select improvements to apply:"

- Options: **"Fix all errors & warnings"** | **"Review each"** | **"Fix errors only"** | **"Skip audit"**

### Step 3.4: Apply Selected Fixes

For each selected improvement: read target file → apply specific fix → show diff → write.

**For skill conversions:** Extract content from rule, create `.sentinal/skills/{slug}-{name}/SKILL.md` (identical content). Remove content from rule, add brief note pointing to skill.

**For file splits:** Create new modular files in `.sentinal/rules/` with the `{slug}-` prefix.

**For conflict resolution:** Present both conflicting instructions, ask user which is correct, update both files.

## Phase 4: Explore Codebase

### Step 4.1: Setup

1. **Quick project overview:** Use `project_context` MCP tool to get tech stack, structure, conventions, and commands in one call.
2. Check `vexor --version` — if available, use Vexor for semantic exploration (Step 4.2). If not installed, use Grep/Glob/Read in Step 4.3.
3. **Directory structure:** `tree -L 3 -I 'node_modules|.git|dist|build|.next|coverage|.cache'`
4. **Technologies:** Check `package.json`, `tsconfig.json`, `angular.json` (or use `project_context` output from step 1)
4. **Source documents:** Find and read canonical docs — `docs/`, `**/ARCHITECTURE.md`, `**/CONTRIBUTING.md`. These are the source of truth for generated rules.

### Step 4.2: Semantic Exploration with Vexor — OPTIONAL

**Skip if Vexor is not installed.** Use `vexor search` (via Bash) to understand each area. **Always use `--max-results 5 --max-tokens 2000`** to keep context lean.

```bash
# Architecture & structure
vexor search "how is the application structured and what are the main entry points" ./ --max-results 5 --max-tokens 2000
vexor search "how are Angular modules and NestJS modules organized" ./ --max-results 5 --max-tokens 2000

# Patterns & conventions
vexor search "how are API endpoints defined in NestJS" ./ --max-results 5 --max-tokens 2000
vexor search "how is authentication and authorization handled" ./ --max-results 5 --max-tokens 2000
vexor search "how is configuration loaded and environment variables used" ./ --max-results 5 --max-tokens 2000
vexor search "how is error handling done" ./ --max-results 5 --max-tokens 2000
vexor search "how are database models or TypeORM entities structured" ./ --max-results 5 --max-tokens 2000

# Angular-specific
vexor search "how are Angular components and services organized" ./ --max-results 5 --max-tokens 2000
vexor search "how is state management handled in Angular" ./ --max-results 5 --max-tokens 2000

# Testing
vexor search "how are tests organized and what testing patterns are used" ./ --max-results 5 --max-tokens 2000

# Build & deploy
vexor search "how is the application built and deployed" ./ --max-results 5 --max-tokens 2000
```

**Follow up with `vexor extract`** to pull concrete examples for rules:

```bash
vexor extract src/routes/users.controller.ts#createUser
vexor extract src/app.module.ts:1-30
```

### Step 4.3: Fill Gaps

**Primary exploration step when Vexor is not available.**

1. **Grep** for key patterns — entry points, route definitions, config loading, error handling, auth, test organization
2. **Glob** for file structure — `**/*.spec.ts`, `**/controllers/**`, `**/services/**`, `**/modules/**`
3. **Read** 5-10 representative files in key directories
4. For each gap from Phase 1: run a targeted search to find current patterns

**Monorepo:** Repeat Steps 4.1-4.3 for each sub-project identified in Phase 1.

## Phase 5: Compare & Identify Gaps

1. For each existing rule: still accurate? new patterns? tech stack changed? commands/paths correct?
2. Identify gaps: undocumented tribal knowledge, new conventions, changed patterns
3. AskUserQuestion to confirm findings: "Update all" | "Review each" | "Show details" | "Skip updates"

## Phase 6: Sync Project Rule

**Update `.sentinal/rules/{slug}-project.md` with current project state.**

Also look for a legacy unscoped `project.md` — if found, migrate its content into `{slug}-project.md` and delete the old file.

### Step 6.1: Handle Existing CLAUDE.md — CONDITIONAL

**Only if Phase 1 found a CLAUDE.md file.**

If both CLAUDE.md AND `{slug}-project.md` exist: merge unique content from CLAUDE.md into the project rule. If fully redundant, suggest removing CLAUDE.md.

If CLAUDE.md exists but NO `{slug}-project.md`: AskUserQuestion:

- **"Migrate to modular rules (Recommended)"** — Split into `{slug}-project.md` + topic-specific files. Read CLAUDE.md, identify logical sections, create rule files, confirm split before writing.
- **"Keep CLAUDE.md as-is"** — Skip project rule creation.
- **"Create alongside"** — Keep both.

### Step 6.2: Create or Update {slug}-project.md

If exists: compare tech stack, verify structure/commands, update timestamp, preserve custom sections.

If doesn't exist, create:

```markdown
# Project: [Name]

**Last Updated:** [Date]

## Overview

[Brief description from README or ask user]

## Technology Stack

- **Language / Framework / Build Tool / Testing / Package Manager**
- NestJS / Angular / TypeScript — include versions from package.json

## Directory Structure

[Simplified tree — key directories only]

## Key Files

- **Configuration / Entry Points / Tests**

## Development Commands

| Task                                | Command     |
| ----------------------------------- | ----------- |
| Install / Dev / Build / Test / Lint | `[command]` |

## Architecture Notes

[Brief patterns description — NestJS modules, Angular standalone/module structure, etc.]
```

**Monorepo / Multi-product:** Create a root `{slug}-project.md` with the overall structure. Then for each product/team area, create nested directories per Phase 0 → Recommended Directory Structure.

### Step 6.3: Generate Rules README — CONDITIONAL

**Only if nested directories exist or are being created in this sync.**

Create or update `.sentinal/rules/README.md`:

```markdown
# Rules Directory Structure

> Auto-generated by `/sync` on [Date].

## How Rules Work

Rules in `.sentinal/rules/` load into Claude's context each session.
Use `paths` frontmatter to scope rules to specific files — unscoped rules load every time.

## Organization

| Directory             | Scope         | Contents                                                    |
| --------------------- | ------------- | ----------------------------------------------------------- |
| `./`                  | Repo-wide     | [List root-level rule files with one-line descriptions]     |
| `./{product}/`        | Product-wide  | [List product rules with descriptions]                      |
| `./{product}/{team}/` | Team-specific | [List team rules with descriptions and their `paths` globs] |

## Adding Rules

1. Identify the narrowest scope: team > product > repo
2. Place file in the corresponding directory
3. Add `paths` frontmatter (**mandatory** for team-level, recommended for product-level)
4. Prefix filename with `{slug}-`
5. Run `/sync` to validate and update this README
```

**README rules:** Auto-generate the table from actual files — don't hardcode. Include each rule's description. Keep under 80 lines.

## Phase 7: Sync MCP Rules

**Document user-configured MCP servers.** Skip Sentinal core servers (context7, mem-search, web-search, web-fetch, grep-mcp) — only document servers the user added themselves.

### Step 7.1: Discover

Parse `.mcp.json`, exclude Sentinal core servers (context7, mem-search, web-search, web-fetch, grep-mcp).

### Step 7.2: Smoke-Test

For each user server:

1. `ToolSearch(query="+server-name keyword")` to discover tools
2. Call each tool with minimal read-only arguments (**safety: only read-only tools**)
3. Record per-tool: success | auth error | connection error | schema error | timeout
4. Report health check:
   ```
   ✅ linear — 3/3 tools working
   ⚠️ jira — 4/5 working, 1 permission error
   ❌ my-api — 0/2 working (connection refused)
   ```
5. If issues: AskUserQuestion "Document working tools only" | "Document all with status notes" | "Skip MCP sync"

### Step 7.3: Document

Compare against existing `{slug}-mcp-servers.md`. If changes detected, ask user: "Update all" | "Review each" | "Skip"

Also look for a legacy unscoped `mcp-servers.md` — if found, migrate content into `{slug}-mcp-servers.md` and delete the old file.

### Step 7.4: Write

Create/update `.sentinal/rules/{slug}-mcp-servers.md`:

```markdown
### [server-name]

**Source:** `.mcp.json`
**Purpose:** [Brief description]
**Status:** ✅ All working | ⚠️ Partial | ❌ Broken

| Tool | Status | Description |
| ---- | ------ | ----------- |

**Example:** `ToolSearch(query="+server-name keyword")` then call directly.
```

**Skip if:** no `.mcp.json`, no user-added servers, user declines.

## Phase 8: Sync Existing Skills

For each skill from Phase 1:

1. **Relevance:** Does the workflow/tool still exist? Has process changed?
2. **Currency:** Steps accurate? APIs changed? Examples working?
3. **Triggers:** Description still accurate for discovery?

If updates needed: AskUserQuestion (multiSelect) with what changed and why. For each selected: update content, bump version (e.g., 1.0.0 → 1.0.1).

If obsolete: AskUserQuestion "Yes, remove it" | "Keep it" | "Update instead". If removing: delete the skill directory.

## Phase 9: Discover New Rules

1. List undocumented areas (comparing Phase 1 + Phase 4)
2. For each candidate area, find actual patterns before drafting:

   ```bash
   # With Vexor (preferred)
   vexor search "how is [pattern] implemented across the codebase" ./ --max-results 5 --max-tokens 2000
   vexor extract src/example.ts#patternFunction

   # Without Vexor (fallback)
   # Grep(pattern="[pattern]", head_limit=10)
   # Read representative files directly
   ```

3. Prioritize by: frequency, uniqueness, mistake likelihood
4. AskUserQuestion (multiSelect): which areas to document
5. For each: ask clarifying questions, draft rule using search results and code examples, confirm before creating
6. **Place in correct directory** based on scope:
   - Repo-wide → `.sentinal/rules/{slug}-{pattern-name}.md`
   - Product-specific → `.sentinal/rules/{product}/{slug}-{product}-{pattern-name}.md` (add `paths` frontmatter)
   - Team-specific → `.sentinal/rules/{product}/{team}/{slug}-{team}-{pattern-name}.md` (**must** have `paths` frontmatter)
   - Create product/team directories as needed (`mkdir -p`)

**Angular/NestJS discovery areas to prioritize:**

- State management patterns (NgRx, Signals, RxJS)
- NestJS module organization and dependency injection
- TypeORM entity conventions and migrations
- Error handling and exception filters
- Tailwind CSS utility class conventions
- Authentication guards and JWT patterns

**Rule format:** Standard Name → When to Apply → The Pattern (code examples) → Why (if not obvious) → Common Mistakes → Good/Bad examples.

## Phase 10: Discover & Create Skills

Skills are appropriate for: multi-step workflows, tool integrations, reusable scripts, domain expertise.

1. Identify candidates from exploration: repeated workflows, complex tool usage, bundled scripts
2. AskUserQuestion (multiSelect): which to create
3. For each: invoke `Skill(skill="learn")` if available — otherwise create `.sentinal/skills/{slug}-{name}/SKILL.md` (identical content) directly with frontmatter (`name`, `description`, optionally `user-invocable: true`)
4. Verify: skill directory exists, SKILL.md has proper frontmatter

## Phase 11: Cross-Check

**Re-read all generated/updated files and verify against source docs and each other.**

1. **Build entity index** — collect all services, entry points, modules, config keys mentioned across generated files
2. **Cross-file completeness** — for each entity, verify it appears in all related files
3. **Source fidelity** — for each identifier in generated rules, confirm exact spelling using Vexor or Grep. Fix rule to match source verbatim if different.
4. **Section coverage** — for each significant section in source docs (CLAUDE.md, `docs/`), confirm a corresponding rule section exists
5. **Reference validity** — cross-references between files point to files that actually exist
6. **README currency** — if `.sentinal/rules/README.md` exists, verify it lists all current rule files and directories. Update if stale.
7. **Path-scoping enforcement** — re-verify all team-level rules have `paths` frontmatter

Auto-fix any issues found. Report fixes in summary.

## Phase 12: Summary

Report:

- Directory structure: flat | nested (list products/teams)
- README: created | updated | not needed (flat structure)
- Quality audit: errors fixed, warnings addressed, suggestions applied, skipped
- Rules: created, updated, unchanged (by directory level)
- Path-scoping: team rules validated, violations fixed
- Skills: created, updated, removed, unchanged
- Cross-check: issues found and fixed (if any)
- Vexor: available / not available (Grep/Glob used as fallback)

Then offer: "Commit `.claude/` to git and push via PR" | "Discover more standards" | "Create more skills" | "Done"
