# TDD Guard Exclusions & Agent Permissions Plan

Status: VERIFIED
Approved: Yes

Type: Feature

## Summary

**Goal:** Stop the TDD guard from blocking edits to NestJS convention files (`.constant.ts`, `.enum.ts`, `.module.ts`, etc.) that are already exempt from companion test warnings, and grant OpenCode's plan and build agents broader subagent/edit permissions so they don't prompt unnecessarily.

**Architecture:** The TDD guard (both Claude Code subprocess and OpenCode in-process plugin) gains an early-return check using the existing `SKIP_TEST_PATTERNS` from `src/utils/tdd.ts`. A new exported helper `shouldSkipTddGuard(filePath)` centralizes this check. OpenCode's `opencode.json` agent permissions are expanded to allow `explore` and `general` subagents, and the build agent gains edit permissions for plan files.

**Tech Stack:** TypeScript, Bun

## Scope

### In Scope

- Add `shouldSkipTddGuard()` helper to `src/utils/tdd.ts` using existing `SKIP_TEST_PATTERNS`
- Apply the helper in `src/hooks/tdd-guard.ts` (Claude Code subprocess)
- Apply the helper in `targets/opencode/plugins/sentinal.ts` (OpenCode in-process guard)
- Expand OpenCode agent permissions in `targets/opencode/opencode.json`:
  - Plan agent: add `explore` and `general` subagent access
  - Build agent: add `explore` and `general` subagent access, plus edit permission for plan files
- Update tests

### Out of Scope

- Claude Code agent permissions (already has `Task(*)` and `Agent` pre-approved — no restrictions to fix)
- Changes to `SKIP_TEST_PATTERNS` list itself (already correct)
- Changes to file-checker / companion test warning system (already uses `SKIP_TEST_PATTERNS`)

  - These same files should not be blocked by the TDD guard because they are tested indirectly through integration tests or other files that import them

## Assumptions

- Files matching `SKIP_TEST_PATTERNS` should NEVER be blocked by the TDD guard — supported by: they are already exempt from companion test warnings (`tdd.ts:23`) and the user explicitly requested this — Tasks 1, 2, 3 depend on this
- OpenCode's `explore` and `general` are valid built-in subagent names for task permission keys — supported by: existing `opencode.json` uses unnamespaced names `plan-reviewer` and `spec-reviewer` as permission keys (not `sentinal:plan-reviewer`), and `explore`/`general` are OpenCode built-in agent types — Task 4 depends on this. If these names don't match OpenCode's resolution, the worst case is a permission prompt (no breakage).
- The plan and build agents benefit from `explore` and `general` subagent access because the spec workflow skills and rules reference `Task(subagent_type="explore")` and `Task(subagent_type="general")` patterns — Task 4 depends on this
