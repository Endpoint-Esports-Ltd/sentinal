## Summary

**Goal:** Reduce per-tool-call latency by ~50-300ms (async hooks), reclaim ~2,000 tokens of context per turn (subagent-scoped MCP), and eliminate post-compaction disorientation (context injection) — with Claude Code and OpenCode parity where possible.

**Architecture:** Three independent optimizations applied to the Sentinal hook system and MCP configuration across both Claude Code (subprocess hooks + plugin config) and OpenCode (in-process plugin callbacks + JSON config). Each option is self-contained and can be verified independently.

**Tech Stack:** TypeScript (Bun), Claude Code hooks API (`async: true`, `UserPromptSubmit`), OpenCode plugin API (`tool.execute.after`, `experimental.session.compacting`, `event`), MCP server configuration (`.mcp.json`, `opencode.json`), subagent frontmatter (`mcpServers` field).

## Scope

### In Scope

- **Option 1:** Make `tdd-tracker`, `memory-observer`, and `context-monitor` hooks async (Claude Code: `"async": true`; OpenCode: fire-and-forget promises)
- **Option 2:** Move 4 external MCP servers (`context7`, `web-search`, `grep-mcp`, `web-fetch`) from main context to subagent scope (Claude Code: subagent `mcpServers` frontmatter; OpenCode: investigate agent-level tool disabling)
- **Option 3:** Add `UserPromptSubmit` hook (Claude Code) and fix `session.created` context injection (OpenCode) to inject active spec state; enrich injected context with current task/progress
- Update embedded assets (`embed-assets`) and installer logic for all config changes
- Tests for the new `UserPromptSubmit` hook and enriched context builder

### Out of Scope

- Conditional rule loading (Tier 1 Option 4 from analysis)
- Dynamic tool schema trimming via OpenCode `tool.definition` hook
- TaskCompleted hook enforcement
- Persistent subagent memory
- Context fork for spec phases
- MCP Tool Counters & Power Management (deferred plan)

## Pre-Mortem

_Assume this plan failed. Most likely internal reasons:_

1. **Async hooks break TDD workflow** (Tasks 1-2) → Trigger: `tdd-tracker` async delivery is too slow, causing `tdd-guard` to see stale state on next edit. Observable: user can edit implementation file without RED*CONFIRMED state showing in time. Mitigated: tdd-tracker writes to SQLite synchronously within the hook process — it's the \_process exit wait* that's removed, not the DB write. The SQLite write completes before the hook exits.

2. **OpenCode MCP scoping doesn't work** (Task 4) → Trigger: OpenCode's `tools` agent config doesn't support MCP wildcard patterns, or subagents inherit parent tool restrictions. Observable: external MCP tools disappear entirely (even in subagents) or remain visible in main context. Mitigated: test with a single server; document gap if unsupported.

3. **UserPromptSubmit context injection is too noisy** (Task 5) → Trigger: injecting spec state on every prompt adds clutter when no spec is active, or when spec state hasn't changed since last injection. Observable: user sees repetitive context. Mitigated: only inject when `findActivePlan()` returns non-null; skip injection if state hash matches previous injection.

### Task 2: Async OpenCode PostToolUse Handlers

**Objective:** Split the OpenCode plugin's `tool.execute.after` handler so that TDD tracking and memory capture run as fire-and-forget promises, while quality checks remain synchronous.

**Dependencies:** None

**Files:**

- Modify: `targets/opencode/plugins/sentinal.ts`

**Key Decisions / Notes:**

- The `tool.execute.after` handler currently runs everything sequentially: context monitoring, quality checks, TDD tracking, memory capture
- Split into two phases:
  1. **Sync phase** (must complete before returning): quality checks (file length, NestJS patterns, companion test) — these can throw Error to block
  2. **Async phase** (fire-and-forget): TDD tracking + memory capture — wrap in `Promise.resolve().then(async () => { ... }).catch(() => {})` so errors are swallowed and the handler returns immediately
- Context monitoring can also move to the async phase since it's informational
- Pattern: `void asyncWork()` — the `void` operator discards the promise (no unhandled rejection)

- [ ] Memory capture runs as fire-and-forget in tool.execute.after
- [ ] Context monitoring runs as fire-and-forget in tool.execute.after

### Task 4: Agent-Level MCP Tool Scoping (OpenCode)

**Objective:** Investigate and implement OpenCode's equivalent of subagent MCP scoping — disable external MCP tools from the main build/plan agents while keeping them available in subagents.

**Dependencies:** None

**Files:**

- Modify: `targets/opencode/opencode.json`
- Potentially modify: `targets/opencode/agents/plan-reviewer.md`
- Potentially modify: `targets/opencode/agents/spec-reviewer.md`

**Key Decisions / Notes:**

- OpenCode's agent `tools` config supports wildcards: `"mcp_context7_*": false` disables all tools from the context7 server
- **Approach A (preferred):** In `agent.build.tools` and `agent.plan.tools`, disable external MCP tools. In subagent definitions (plan-reviewer, spec-reviewer), re-enable them with `"mcp_context7_*": true` etc.
- **Approach B (fallback):** If wildcards don't work for MCP tools or subagents don't re-inherit, document the gap and skip OpenCode changes.
- Key question: do OpenCode subagents inherit tool restrictions from their parent agent? The docs say "subagents will use the model of the primary agent that invoked the subagent" for model but tool inheritance is less clear.
- If subagents DO inherit tool restrictions and can't override them, this approach won't work — document as known limitation.

**Implementation Result: GAP DOCUMENTED**

OpenCode does not support per-agent MCP server scoping equivalent to Claude Code's subagent `mcpServers` frontmatter. While OpenCode agents support a `tools` config with wildcards (e.g., `mcp_context7_*: false`), there are two problems:

1. **Subagent tool inheritance is unclear** — the docs don't specify whether subagents inherit tool restrictions from their parent agent. If they do and can't override, disabling MCP tools at the build/plan level would also remove them from subagents (plan-reviewer, spec-reviewer, explore).
2. **No inline MCP server definitions for subagents** — OpenCode subagent markdown files don't support a `mcpServers` field. MCP servers are configured globally in `opencode.json`.

This is a known platform gap. External MCP servers remain in the global `mcp` section of `opencode.json` for OpenCode. The Claude Code implementation (Task 3) provides the full optimization there.

**Definition of Done:**

- [x] Gap documented in plan file as known limitation with explanation
- [x] No OpenCode config changes made (preserving working state)

### Task 6: Enriched Compaction + Session Context (OpenCode)

**Objective:** Enhance the OpenCode plugin's compaction handler to include current task and progress information, and fix session.created to inject memory context into LLM context (not just TUI log).

**Dependencies:** None

**Files:**

- Modify: `targets/opencode/plugins/sentinal.ts`

**Key Decisions / Notes:**

- **Compaction enrichment:** The current `experimental.session.compacting` handler pushes plan path and status into `output.context`. Enhance to also include: current task name, progress percentage, remaining task count. Use `SpecStore.getCurrentTask()` via sidecar or parse the plan file directly.
- **Session.created fix:** Currently, `session.created` calls `sidecar.restoreContext()` and logs it via `client.app.log()` — which only shows in TUI panel, NOT in LLM context. - **Context builder duplication (not sharing):** The OC plugin is deployed as a single bundled file and cannot import from `src/`. Duplicate the ~10 lines of context formatting logic inline in the OC plugin's compaction handler. Add cross-reference comments in both locations (`src/hooks/prompt-context.ts` and `targets/opencode/plugins/sentinal.ts`) to keep them in sync. This matches the existing codebase pattern where the OC plugin duplicates small utilities.
