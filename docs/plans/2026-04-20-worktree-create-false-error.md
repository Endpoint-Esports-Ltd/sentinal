# worktree_create Reports False Error Despite Successful Git Worktree Creation

Created: 2026-04-20
Status: VERIFIED
Approved: Yes
Iterations: 0
Worktree: No
Type: Bugfix

## Summary

**Symptom:** `worktree_create` MCP tool returns "Error creating worktree: FOREIGN KEY constraint failed". The AI agent interprets this as "Sentinal sidecar issue - fallback to git directly". But `git worktree list` shows the worktree was actually created on disk.

**Trigger:** Calling `worktree_create` with a `plan_slug` that hasn't been registered in the `specs` table via `spec_register` yet. This is the normal workflow — the spec plan skills (spec-plan, spec-bugfix-plan) create the worktree (Step 1.1.2) BEFORE writing the plan header (Step 1.1.5) and registering the spec (Step 1.1.6).

**Root Cause:** `src/worktree/manager.ts:90-96` — Two operations execute sequentially without transactional safety:
1. Line 90-93: `gitExecOrThrow()` creates the git worktree on disk (succeeds, side-effect committed)
2. Line 96-106: `this.store.insert()` attempts SQLite INSERT with `spec_id` referencing `specs(id)` FK
3. If the spec isn't registered yet, the FK constraint at `src/memory/migrations.ts:273` (`spec_id TEXT REFERENCES specs(id)`) fires and the INSERT fails
4. The error propagates to the MCP tool's catch block (`mcp-tools.ts:127-129`) which returns `mcpError("Error creating worktree", err)`
5. **No cleanup**: the git worktree remains on disk despite the error

## Investigation

### Call graph traced

```
worktree_create MCP tool (mcp-tools.ts:116)
  → manager.create(plan_slug, projectPath, base_branch)  (manager.ts:43)
    → gitExecOrThrow(["worktree", "add", ...])            (manager.ts:90) ← git side-effect HERE
    → store.insert({specId: plan_slug, ...})              (manager.ts:96)
      → INSERT INTO worktrees (spec_id=plan_slug, ...)    (store.ts:39-54)
        → FK check: specs(id) has plan_slug?              (migrations.ts:273)
        → NO → FOREIGN KEY constraint failed              ← ERROR HERE
```

### Why the worktree is "already created"

`gitExecOrThrow()` at line 90 runs `git worktree add -b <branch> <path> <base>`, which succeeds immediately. The git operation is not transactional with the SQLite insert — if the insert fails, the worktree on disk is orphaned.

### FK constraint details

- `migrations.ts:273`: `spec_id TEXT REFERENCES specs(id)` — hard FK
- `store.ts:60`: `PRAGMA foreign_keys = ON` — enforced on every `MemoryStore` connection
- `store.ts:46`: `wt.specId ?? null` — NULL would bypass FK, but `plan_slug` is always a string from the MCP tool's zod schema (`z.string()`), so it's never null

### The "sidecar issue" error message is NOT from Sentinal

The string "Sentinal sidecar issue - fallback to git directly" does not exist anywhere in the sentinal codebase. It's generated by the AI agent when it receives the MCP error and attempts to explain/recover. The actual error from the tool is `"Error creating worktree: FOREIGN KEY constraint failed"`.

### Workflow ordering creates the race

The spec plan skills instruct (Step 1.1):
1. Step 1.1.2: Create worktree (`worktree_create` MCP tool) ← `spec_id` not in DB yet
2. Step 1.1.5: Write plan file header
3. Step 1.1.6: Register spec (`spec_register` MCP tool) ← `spec_id` enters DB here

The worktree creation always happens before spec registration. This is by design (the plan file should be written into the worktree), but it conflicts with the FK constraint.

## Behavior Contract

### Fix Property (C => P)

**When `worktree_create` is called with a `plan_slug` not yet registered in `specs`:**
The worktree is created successfully on disk AND recorded in SQLite. No error is returned.

### Preservation Property (!C => unchanged)

**When `worktree_create` is called with a `plan_slug` that IS registered in `specs`:**
Existing behavior preserved — worktree created and linked to spec.

## Fix Approach

**Strategy:** Defer `spec_id` linkage. Insert the worktree row with `spec_id = NULL` (which bypasses the FK constraint), then provide a way to link it later.

**Files:**

| File | Change |
|------|--------|
| `src/worktree/manager.ts` | In `create()`: always insert with `specId = undefined` (→ NULL). Store the specId on the returned Worktree object but don't persist it until linkage. Add `linkSpec(worktreeId, specId)` method. Add try/catch around `store.insert()` with git worktree cleanup on failure. |
| `src/worktree/store.ts` | Add `updateSpecId(id, specId)` method. |
| `src/worktree/mcp-tools.ts` | Add return of the worktree ID in `worktree_create` response so callers can link later. |

**Why not remove the FK constraint?** Removing FKs in SQLite requires table recreation (new migration). The FK is useful for data integrity when the spec exists. The fix is to defer linkage, not remove the constraint.

**Why not reorder the workflow?** Changing all spec plan skills to register specs before creating worktrees would fix the symptom but is fragile — any new skill or manual MCP call would hit the same bug. The tool itself should be robust.
