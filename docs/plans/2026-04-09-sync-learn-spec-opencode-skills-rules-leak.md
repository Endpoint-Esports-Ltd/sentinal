## Summary

**Symptom:** Running `/sync`, `/learn`, or other `/spec` workflow commands inside an OpenCode (or Claude Code) session results in skill and rule files being created under `.opencode/skills/` and `.opencode/rules/` (and occasionally `.claude/skills/` / `.claude/rules/`) instead of the canonical `.sentinal/skills/` and `.sentinal/rules/`.

**Trigger:**
- Running `/learn` in any project where the installer's `.opencode/skills → ../.sentinal/skills` symlink is absent or broken → writes land in a real `.opencode/skills/<name>/SKILL.md`.
- Running `/sync` in any project → the stale `templates/commands/sync.md` instructs the agent to read/write `.claude/rules/project.md`.
- Fresh `sentinal install` in any user project → `targets/opencode/AGENTS.md` and `src/cli/commands/install-constants.ts` inject `.opencode/rules/standards-*.md` references into the user's AGENTS.md.
- Anyone running `node scripts/generate-commands.js` → regenerates `targets/claude-code/commands/*.md` and `targets/opencode/commands/*.md` from the stale templates, wiping out the correct `.sentinal/`-based content.

**Root Cause:** Six locations tell the agent (or install process) to use legacy `.opencode/` or `.claude/` paths instead of `.sentinal/`, PLUS a structural divergence between `templates/commands/` (stale stubs) and `targets/{claude-code,opencode}/commands/` (hand-edited authoritative files). The generator script is a loaded gun pointed at the fix.

**Diff analysis (post-investigation):**

The only systematic per-target variation is frontmatter (`user-invocable: true`, `model: opus/sonnet`). The remaining bodies are either identical or diverged through independent hand-editing. A Mustache/Handlebars generator would add pipeline complexity for near-zero return. **Decision: delete `scripts/generate-commands.js` and `templates/commands/`, keep target files as the source of truth. Do NOT create opencode command equivalents for the spec sub-phases — they are deliberately skills.**

**OpenCode architecture (from `bc11bf1` / plan `2026-03-10-opencode-agents-skills.md`):** `/spec`, `/sync`, and `/learn` are the only user-invocable commands. The spec sub-phases (`spec-plan`, `spec-implement`, etc.) are skills in `targets/opencode/skills/` — invoked programmatically by the `/spec` dispatcher via the `skill` tool, never typed by the user. This was an explicit architectural decision: "Commands and skills coexist in OpenCode. The dispatcher (`spec`) stays as a command … and the sub-phases become skills." This is still the right design — do not add back command files for the sub-phases.

The no-leak test must therefore also scan `targets/opencode/skills/*/SKILL.md` for forbidden substrings.

## Investigation

### Backward trace from symptom

1. User sees `.opencode/skills/<slug>-<name>/SKILL.md` created after `/learn` → the instruction to write there must come from somewhere the agent loads.
2. OpenCode loads slash commands from `~/.config/opencode/command/` (shipped via `targets/opencode/commands/`) — sentinal's CURRENT `targets/opencode/commands/learn.md` is CORRECT (references `.sentinal/skills/{slug}-{name}/SKILL.md`, line 45, 67, 163).
3. But `templates/commands/learn.md:38-40` — the source template — still says `.claude/skills/` and `.opencode/skills/`. Running the generator would re-introduce the bug.
4. Additionally, the user's issue says "previous session" investigated this — likely saw `.opencode/skills/*` artifacts from a prior `/learn` run that happened *before* the target files were hand-fixed (git blame on `targets/opencode/commands/learn.md` shows 18 Mar 15:27 last edit).
5. For `/sync`: `templates/commands/sync.md` is a 27-line stub with hardcoded `.claude/rules/` — any project where the user runs `sentinal install` after running the generator gets the broken version.

### Working example for comparison

`targets/opencode/commands/sync.md:38-44` (the hand-edited correct version):
```md
### Output Locations
**Custom rules** in `.sentinal/rules/`: `{slug}-project.md` (tech stack, structure), `{slug}-mcp-servers.md` (custom MCP servers), `{slug}-{pattern-name}.md` (tribal knowledge).
**Custom skills** in `.sentinal/skills/`: `{slug}-{name}/SKILL.md`.
Both are read by Claude Code and OpenCode via symlinks and config (set up by `sentinal install`).
```

And `src/cli/commands/install.ts:804-807` confirms OpenCode reads rules by adding `.sentinal/rules/*.md` to the `instructions` array in `opencode.json`:
```ts
const instructions = (config.instructions as string[]) ?? [];
if (!instructions.includes(".sentinal/rules/*.md")) {
  config.instructions = [...instructions, ".sentinal/rules/*.md"];
}
```

And `install.ts:930-934` creates the symlink pairs, including `.opencode/skills → ../.sentinal/skills`, which is why writes to `.opencode/skills/` transparently land in `.sentinal/skills/` on correctly-installed projects.

### Why `src/project/context.ts:56` stays

`RULES_GLOBS = [".claude/rules", ".opencode/rules"]` in `src/project/context.ts:56` is the `project_context` MCP tool scanning for rules when rendering the project summary. This is legacy backward-compat reading and must remain — but it should ALSO include `.sentinal/rules` (it doesn't currently, which is a separate minor enhancement, not in scope).

### Why the `Phase 0.5 Migrate Legacy` blocks stay

`targets/{claude-code,opencode}/commands/sync.md:156-173` contains 18 lines of shell-script-style migration logic that legitimately references `.claude/rules/`, `.claude/skills/`, `.opencode/rules/`, `.opencode/skills/`. This is intentional — the migration phase has to look for legacy paths to move them. **These references MUST NOT be changed.** The lint test needs a whitelist for this block.

## Behavior Contract

### Fix Property (C => P)

**When condition C holds:** A developer or user runs `/sync` or `/learn`, OR runs `sentinal install` in a user project.

**Property P must hold:** No file created, modified, or injected by any of these actions contains a path reference of the form `.opencode/skills/<x>`, `.opencode/rules/<x>`, `.claude/skills/<x>`, or `.claude/rules/<x>` as a destination for new content (not as a legacy-detection probe). All destinations for newly-created rules and skills MUST be `.sentinal/rules/...` and `.sentinal/skills/...`.

### Preservation Property (!C => unchanged)

**When condition C does NOT hold:**
1. The legacy-migration blocks in `targets/{claude-code,opencode}/commands/sync.md:156-173` and `:205-207` continue to detect and migrate `.claude/rules/`, `.claude/skills/`, `.opencode/rules/`, `.opencode/skills/` as before — unchanged.
2. `src/project/context.ts:56` `RULES_GLOBS` continues to read legacy paths for backward compat — unchanged.
3. `src/cli/commands/install.ts:893-977` `setupProjectSymlinks()` continues to create `.claude/rules`, `.claude/skills`, `.opencode/skills` symlinks pointing at `.sentinal/` — unchanged.
4. All other command content (the 500+ hand-edited lines in `targets/*/commands/sync.md` and `learn.md`) — unchanged.
5. `bun test` existing suite continues to pass — no regressions.

## Fix Approach

**Strategy:** Three prongs — (1) delete the generator + stale templates, create missing opencode command files, (2) strip `.opencode/rules/` references from shipped user-project AGENTS.md templates and team-sharing docs, (3) prevent re-regression via a permanent lint test that runs in `bun test`.

### Prong 1: Delete generator + templates

The `scripts/generate-commands.js` script and `templates/commands/` directory are dead code. The generator's only per-target transformation is frontmatter fields (`user-invocable: true`, `model: opus/sonnet`) — not worth a pipeline. The target files in `targets/{claude-code,opencode}/commands/` and `targets/opencode/skills/` are the source of truth.

**OpenCode architecture is correct and unchanged:** `/spec`, `/sync`, `/learn` are the three user-invocable commands. The spec sub-phases are skills (`targets/opencode/skills/`), not commands. No new command files are needed.

**Delete:**
- `scripts/generate-commands.js`
- `templates/commands/` (entire directory — 8 stale stub files)

**Update `.sentinal/rules/sentinal-targets-vs-src.md`** (the rule file that currently says "Commands come from templates … hand-editing the generated files will be overwritten") — replace the "Commands come from templates" section with accurate statement: "Target command files are the canonical source; there is no generator."

### Prong 2: Fix shipped user-project templates

- `targets/opencode/AGENTS.md` lines 40-46: change `.opencode/rules/standards-*.md` → `.sentinal/rules/standards-*.md`.
- `src/cli/commands/install-constants.ts` lines 86, 97: change `.opencode/rules/` → `.sentinal/rules/`.
- `targets/claude-code/rules/team-sharing.md` lines 23-33: change `.claude/rules/<name>.md` / `.claude/skills/<name>/SKILL.md` → `.sentinal/rules/<name>.md` / `.sentinal/skills/<name>/SKILL.md`.
- `targets/opencode/rules/team-sharing.md` lines 23-33: same.
- After all edits to `targets/*` and `src/cli/commands/install-constants.ts`, re-run `bun run scripts/embed-assets.mjs` to regenerate `src/cli/embedded-assets.ts`.

### Prong 3: Regression lint test

Create a `no-leak` test that scans known source-of-truth files for FORBIDDEN substrings. The substring list is narrow enough to skip legacy-migration shell blocks:

```ts
const FORBIDDEN = [
  ".opencode/skills/{",           // template telling agent where to WRITE <name>
  ".opencode/rules/standards-",   // shipped user standards paths
  ".opencode/rules/{",
  ".claude/skills/{",
  ".claude/rules/{",
  ".claude/skills/<name>",
  ".claude/rules/<name>",
];
```

The legacy-detection shell block in `targets/*/commands/sync.md:156-173` uses `".claude/rules"` and `".opencode/rules"` without `/{` or `/<name>` suffixes → does not match any FORBIDDEN substring → legitimately clean. Verified by hand.

**Test files:**
1. **`src/cli/commands/no-leak.test.ts`** — reads every `targets/*/commands/*.md`, every `targets/opencode/skills/*/SKILL.md`, every `targets/*/AGENTS.md`, every `targets/*/rules/team-sharing.md`, and asserts none contain FORBIDDEN substrings. Also asserts `templates/commands/` directory does NOT exist (generator deleted).
2. **`src/cli/commands/install-constants.test.ts`** — imports `AGENTS_MD_LOCAL_TEMPLATE` and `AGENTS_MD_APPEND` and asserts they do NOT contain `.opencode/rules/`.
