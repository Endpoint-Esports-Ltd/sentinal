## Summary

**Goal:** Leverage two new OpenCode v1.3.x capabilities: (1) adopt `experimental.chat.system.transform` hook for injecting active spec state and model routing hints into every LLM call, and (2) audit/simplify async workarounds now that v1.3.5 fixed async plugin hooks.
**Architecture:** Add a new hook handler in `sentinal.ts` that intercepts the system prompt array to inject spec context (active plan, current task, progress). Also audit fire-and-forget patterns and session creation workarounds for unnecessary complexity.
**Tech Stack:** TypeScript, OpenCode plugin SDK (`@opencode-ai/plugin`), sidecar client

## Scope

### In Scope

- Add `experimental.chat.system.transform` hook to inject active spec state into LLM context
- Add sidecar `GET /config/model-routing` endpoint to expose model routing config (plugin can't use `bun:sqlite` directly)
- Inject model routing hints via `experimental.chat.system.transform` when config differs from defaults
- Audit async hook patterns in sentinal.ts — remove workarounds that v1.3.5 makes unnecessary
- Simplify eager session creation fallback if `session.created` now fires reliably

### Out of Scope

- Dynamic model switching (no `chat.model` hook exists — PR #18826 closed)
- OpenCode `opencode.json` agent model patching (static, restart required)
- TUI plugins / `tui.prompt.append` (undocumented, premature)
- Prompt slot feature (undocumented API, no clear benefit over system.transform)

## Context for Implementer

- **Patterns to follow:** `experimental.session.compacting` handler in `sentinal.ts:636-706` — already injects spec context into compaction output using `output.context.push()`
- **Conventions:** OpenCode plugin is a single file at `targets/opencode/plugins/sentinal.ts`. All hook handlers are in the returned hooks object.
- **Key files:**
  - `targets/opencode/plugins/sentinal.ts` — main plugin file (912 lines)
  - `src/spec/detect.ts` — `findActivePlan()` returns active spec state
  - `src/config/model-routing.ts` — `resolveModelRouting()`, `getModelRouting()`
  - `src/hooks/prompt-context.ts` — Claude Code's equivalent prompt injection (reference)
- **Gotchas:**
  - `sentinal.ts` is at 912 lines — well over the 600-line limit. Must not increase significantly. The system.transform handler should be compact.
  - `experimental.chat.system.transform` has a safety fallback: if plugin empties the array, original is restored
  - The hook receives the system prompt as an array of entries — we should push new entries, not replace existing ones
  - Memory context is currently injected only during compaction (`experimental.session.compacting`). The new hook would inject spec context on EVERY turn, not just compaction.

## Assumptions

- `experimental.chat.system.transform` hook receives `(input, output)` where `output` contains a mutable system prompt array — supported by [prompt construction gist](https://gist.github.com/rmk40/cde7a98c1c90614a27478216cc01551f) — Task 1 depends on this
- The hook is called on every LLM invocation (not just first turn) — supported by its position in the prompt assembly pipeline — Task 1 depends on this
- v1.3.5 "Fixed plugin hooks for async operations" means our fire-and-forget patterns are now handled correctly — needs verification — Task 2 depends on this

### Task 1: Add `experimental.chat.system.transform` hook

**Objective:** Add a handler that injects active spec state and model routing hints into the system prompt array on every LLM call.
**Dependencies:** Task 0
**Wave:** 1

**Files:**

- Modify: `targets/opencode/plugins/sentinal.ts`

**Key Decisions / Notes:**

- **First:** Update the local `PluginHooks` interface (lines 97-118) to declare `"experimental.chat.system.transform"` with signature `(input: Record<string, unknown>, output: Record<string, unknown>) => Promise<void>`. Use `Record<string, unknown>` for both — the actual property names are discovered at runtime via the output shape probe.
- Add `"experimental.chat.system.transform": async (input, output) => { ... }` to the hooks object
- **Output shape probe (gate):** Before writing any injection logic, log `Object.keys(output)` to discover actual property names. Do NOT write injection code until the output shape is confirmed. If shape doesn't match expectations, log a warning and return early.
- When an active spec plan exists (via `findActivePlan(projectRoot)`): push a context entry like `"[Sentinal] Active plan: <name> | Status: <status> | Task: <current> | Progress: <n>/<total>"`
- When model routing differs from defaults: push `"[Sentinal] Model routing: planning=<x>, implementation=<y>. Use the specified model when invoking spec skills."`
- Wrap entire handler in try/catch — system prompt injection is supplementary, must never break the LLM call
- **Tech debt note:** This creates a third copy of spec context formatting logic (alongside compaction handler + `prompt-context.ts`). Track as future refactor to extract shared `formatSpecContext()` utility.
- Keep the handler under 30 lines — extract helpers if needed
- Note: sidecar may not be available for spec detection; use `findActivePlan(projectRoot)` directly (doesn't need sidecar). `projectRoot` comes from the plugin's captured `project` context (same as used elsewhere in the plugin — search for `projectRoot` variable)
- For model routing: call sidecar `GET /config/model-routing` (added in Task 0) to fetch routing config without `bun:sqlite`. Compare with defaults; if different, push a hint entry.

## Async Audit Results

**Finding: All 4 patterns are still necessary. No simplifications are safe.**

The v1.3.5 fix improved the _framework's_ ability to await hook callbacks, but our workarounds address _timing/ordering_ issues between session events and sidecar availability, not async framework bugs.

| Pattern                      | Lines            | Finding                                                                            | Action |
| ---------------------------- | ---------------- | ---------------------------------------------------------------------------------- | ------ |
| Eager session creation       | 349-362          | Still needed — `session.created` may not fire for resume (`opencode -s <id>`)      | Keep   |
| Fire-and-forget TDD tracking | 230-274          | Intentionally fire-and-forget (non-blocking by design) — not a workaround          | Keep   |
| Observation queue drain      | 327-340, 741-747 | Sidecar timing gap — early observations need queuing regardless of async fixes     | Keep   |
| session.updated fallback     | 896-930          | Paired with eager creation — replaces placeholder ID with real one when it arrives | Keep   |
